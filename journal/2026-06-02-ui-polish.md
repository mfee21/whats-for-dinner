## Why these choices

**Search in RecipeIndex, local state only:** The search query is ephemeral UI state â€” no reason to lift it to the parent. Keeping it inside the component means neither `HomePage` nor `RecipesPage` need to know about it, and the component stays a self-contained unit.

**When search is active, replace alpha-nav with a flat list:** Mixing the letter-jump UI with search results would be confusing â€” two navigation modes in one sidebar. Swapping them entirely is cleaner and avoids a stale "active letter" state interfering with search results.

**Render-time ingredient sanitization in addition to parser fix:** The parser fix only helps new recipes. Existing stored data already has the `â–˘` character baked in. Stripping it at render time in `RecipeCookPage` means both old and new recipes look correct immediately, with no forced DB migration. The SQL snippet is provided for users who want to clean the source data optionally.

**Padding on children instead of the article:** With `py` on the article, the tile height is determined by the taller child, but the `py` space exists around both children. If the right child (image + button) is shorter, the bottom padding hangs below it. Moving `py` to each child directly ties the tile border to each child's own content boundaries â€” equal spacing is guaranteed for the right side regardless of left side text length.

**Full-tile click area via pseudo-element overlay:** A common, accessible card-link pattern. The `::after` pseudo-element is `position: absolute` (no layout impact), covers the whole card, and lets the photo upload label sit above it with `z-10`. No JS, no wrapping the whole card in an anchor (which would nest interactive elements inside each other).
